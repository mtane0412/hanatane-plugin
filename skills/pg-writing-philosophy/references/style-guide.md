# 文体とトーンのガイド

このドキュメントは、ポール・グレアム風の文体とトーンを実現するための具体的なガイドラインを提供します。

## 文体の基本原則

### 1. 短い文を使う

ポール・グレアムの文は短いです。平均的に15-20語程度です。

**目安**:
- 短い文: 10語以下
- 標準的な文: 15-20語
- 長い文: 25語以上(避けるべき)

**悪い例(40語)**:
```
スタートアップを成功させるためには、優れたプロダクトを作り、ユーザーを理解し、素早く改善していくことが重要であり、そのためにはチーム全体が同じ目標に向かって協力する必要がある。
```

**良い例(各文10-15語)**:
```
スタートアップを成功させるには何が必要か? まず優れたプロダクトだ。次にユーザーの理解だ。そして素早い改善だ。これにはチームの協力が不可欠だ。
```

### 2. 短い段落を使う

段落も短くしてください。1段落は3-5文が理想です。

**段落の長さの目安**:
- 1行: 強調したいポイント
- 2-3文: 標準的な段落
- 4-5文: やや長い段落
- 6文以上: 長すぎる(分割を検討)

**悪い例(1段落に8文)**:
```
スタートアップは難しい。多くの人が失敗する。なぜだろうか? 理由はいくつかある。まずプロダクトが悪い場合だ。次に市場がない場合だ。また資金が尽きることもある。しかし最大の理由は、諦めることだ。
```

**良い例(3段落に分割)**:
```
スタートアップは難しい。多くの人が失敗する。なぜだろうか?

理由はいくつかある。プロダクトが悪い。市場がない。資金が尽きる。

しかし最大の理由は、諦めることだ。
```

### 3. 単純な語彙を使う

難しい言葉よりも、誰でも知っている簡単な言葉を使ってください。

**難しい言葉 → 簡単な言葉**:
- 利用する → 使う
- 実施する → する、やる
- 検討する → 考える
- 把握する → わかる、知る
- 保有する → 持つ
- 発生する → 起きる
- 認識する → 気づく、わかる
- 判断する → 決める
- 獲得する → 得る、手に入れる

**例**:
```
悪い: 我々は新しい戦略を検討し、実施することを決定した
良い: 私たちは新しい戦略を考え、やることに決めた
```

## トーンの原則

### 1. カジュアルだが誠実

友人と話すようにカジュアルに書いてください。しかし、軽薄にならないように注意してください。誠実さを保ってください。

**バランス**:
- 堅苦しすぎる ← **適切** → 軽薄すぎる

**堅苦しすぎる例**:
```
本稿において論述する内容は、スタートアップの成長戦略に関する考察である。
```

**適切な例**:
```
スタートアップの成長戦略について考えてみよう。
```

**軽薄すぎる例**:
```
スタートアップの成長戦略ってヤバいよね!笑
```

### 2. 自信を持つが傲慢にならない

自分の意見を自信を持って述べてください。しかし、傲慢にならないように注意してください。

**良いバランス**:
```
私はこう考えている: スタートアップは小さく始めるべきだ。もちろん、これに反対する人もいるだろう。しかし私の経験では、小さく始めた方がうまくいくことが多い。
```

**傲慢な例**(避けるべき):
```
スタートアップは小さく始めるべきだ。これに反対する人は間違っている。私が正しい。
```

**自信がなさすぎる例**(避けるべき):
```
私の考えでは、もしかしたら、スタートアップは小さく始めた方がいいかもしれません。ただの個人的な意見ですが...
```

### 3. 読者を尊重する

読者を見下さないでください。読者が理解できないことを前提にしないでください。

**良い例**:
```
APIとは何か? 簡単に言えば、プログラム同士が通信する方法だ。
```

**悪い例**(見下している):
```
APIという言葉を知らない人もいるかもしれないので、初歩から説明しよう。
```

## 文のパターン

### 1. 疑問文で始める

段落や新しいトピックを疑問文で始めるのは効果的です。

**パターン**:
```
[疑問文]? [答え]。[詳細な説明]。
```

**例**:
```
なぜスタートアップは失敗するのか? 最大の理由は、誰も欲しがらないものを作るからだ。これは驚くべきことに、最も一般的な失敗の原因だ。
```

### 2. 短い答えで強調する

疑問文の後に、非常に短い答えを置くと効果的です。

**パターン**:
```
[疑問文]? [1-3語の答え]。
```

**例**:
```
スタートアップに必要なものは何か? 成長だ。

良いアイデアの条件とは? 驚きだ。

プログラマーが最も重視すべきものは? シンプルさだ。
```

### 3. 対比を使う

対比は、ポイントを明確にする優れた方法です。

**パターン**:
```
[一般的な考え]。しかし実際は[真実]。
```

**例**:
```
多くの人は、スタートアップには大きなビジョンが必要だと考えている。しかし実際は、小さな問題を解決することから始まることが多い。
```

### 4. 具体例で始める

抽象的な主張の前に、具体例を置くことも効果的です。

**パターン**:
```
[具体例]。これは[抽象的な原則]を示している。
```

**例**:
```
Airbnbは家賃を払えないことから始まった。Stripeはオンライン決済が複雑すぎることから始まった。これは、優れたプロダクトは小さな問題から始まることを示している。
```

## 避けるべき表現

### 1. 冗長な表現

以下の表現は削除してください:

| 冗長な表現 | 簡潔な表現 |
|----------|----------|
| ～することができる | ～できる |
| ～と言える | (削除) |
| ～と考えられる | ～と思う |
| ～の場合 | ～なら |
| ～に関して | ～について |
| ～という事実 | (削除) |
| ～ということ | (削除) |
| 基本的に | (削除) |
| 実際のところ | 実際は |

**例**:
```
悪い: 基本的に、スタートアップが成功することができるかどうかは、プロダクトの質に関して依存すると考えられる
良い: スタートアップの成功は、プロダクトの質で決まる
```

### 2. 受動態

可能な限り能動態を使ってください。

**受動態 → 能動態**:
```
受動: この問題は解決された
能動: 私たちはこの問題を解決した

受動: 多くの企業によって採用されている
能動: 多くの企業が採用している
```

### 3. 名詞化

動詞を名詞化すると、文章が固くなります。動詞のまま使ってください。

**名詞化 → 動詞**:
| 名詞化 | 動詞 |
|-------|------|
| 実施する | する、やる |
| 検討する | 考える |
| 提供する | 提供する(そのまま) |
| 利用する | 使う |
| 確認する | 確かめる、見る |

**例**:
```
悪い: プロダクトの改善を実施する
良い: プロダクトを改善する
```

### 4. 曖昧な数量詞

「多くの」「いくつかの」「最近」などの曖昧な表現は避け、具体的な数字や時期を使ってください。

**曖昧 → 具体的**:
```
曖昧: 多くのスタートアップが失敗する
具体的: 90%のスタートアップが失敗する

曖昧: 最近のトレンド
具体的: 2023年のトレンド

曖昧: いくつかの理由
具体的: 3つの理由
```

## 声に出して読む

最も重要なテストは、声に出して読むことです。

**チェックポイント**:
1. つかえずに読めるか?
2. 自然に聞こえるか?
3. 友人との会話のように聞こえるか?
4. 呼吸が苦しくならないか?(文が長すぎる場合、呼吸が苦しくなる)

声に出して読んで不自然な部分は、書き直してください。

## リズムとテンポ

### 1. 文の長さを変化させる

すべての文が同じ長さだと、単調になります。長い文と短い文を混ぜてください。

**例**:
```
スタートアップは難しい。(短)
多くの人が失敗する理由は、プロダクトが悪いからだ。(中)
しかし最大の理由は諦めることだ。(中)
諦めるな。(極短)
```

### 2. パンチラインを短くする

最も重要なポイント(パンチライン)は、短い文で表現してください。

**例**:
```
スタートアップを成功させるには、優れたプロダクトが必要だ。ユーザーを理解する必要がある。素早く改善する必要がある。しかし最も重要なのは何か?

諦めないことだ。
```

### 3. 繰り返しを使う

同じ構造を繰り返すと、リズムが生まれます。

**例**:
```
良いプロダクトは小さく始まる。
良いプロダクトはシンプルだ。
良いプロダクトは実際の問題を解決する。
```

## まとめ: PG風文体のチェックリスト

文章を書いた後、以下をチェックしてください:

- [ ] 文は短いか?(平均15-20語)
- [ ] 段落は短いか?(3-5文)
- [ ] 簡単な言葉を使っているか?
- [ ] 会話調になっているか?
- [ ] 疑問文を使っているか?
- [ ] 具体例があるか?
- [ ] 冗長な表現を削除したか?
- [ ] 受動態を能動態に変換したか?
- [ ] 声に出して読んで自然か?
- [ ] リズムとテンポがあるか?

これらすべてがYesなら、PG風の文体になっています。
